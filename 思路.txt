可以。那这个项目就不要设计成“普通 textarea 便签”，而应该设计成：

> **Windows 顶部隐藏式 Markdown 素材便签抽屉**

核心能力是：

```text
鼠标靠近屏幕顶部 → 展开抽屉
在抽屉里写 Markdown
可以粘贴图片 / 拖拽图片 / 从本地添加图片
内容和图片都保存在本地
再次打开时完整恢复
```

技术上我建议用：

```text
Tauri v2 + React + TypeScript + TailwindCSS + Milkdown/Crepe + SQLite + 本地图片目录
```

Tauri 负责窗口、文件、SQLite、系统能力；React 负责 UI；Milkdown/Crepe 负责 Markdown 编辑器；SQLite 存便签元数据和 Markdown 内容；图片复制到应用数据目录保存。Tauri v2 有官方文件系统插件、Dialog 插件和 SQL 插件，适合这个结构；Milkdown 的 Crepe 编辑器本身支持更完整的 Markdown/WYSIWYG 编辑体验，并包含图片块、图片上传管理、调整尺寸、caption 等能力。([Tauri][1])

# 2. 核心数据模型

## notes 表

```sql
CREATE TABLE notes (
  id TEXT PRIMARY KEY,
  title TEXT NOT NULL,
  content TEXT NOT NULL,
  excerpt TEXT,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  pinned INTEGER NOT NULL DEFAULT 0,
  deleted INTEGER NOT NULL DEFAULT 0
);
```

说明：

```text
id：便签唯一 ID
title：标题，默认取 Markdown 第一行
content：完整 Markdown 内容
excerpt：摘要，用于列表展示
created_at：创建时间
updated_at：更新时间
pinned：是否置顶
deleted：软删除
```

---

## assets 表

```sql
CREATE TABLE assets (
  id TEXT PRIMARY KEY,
  note_id TEXT,
  type TEXT NOT NULL,
  file_name TEXT NOT NULL,
  file_path TEXT NOT NULL,
  mime_type TEXT NOT NULL,
  size INTEGER NOT NULL,
  created_at INTEGER NOT NULL,
  FOREIGN KEY(note_id) REFERENCES notes(id)
);
```

说明：

```text
assets 用来管理图片资源
每张图片都有独立 ID
图片文件存在本地目录
Markdown 内容里通过 asset id 或相对路径引用图片
```

# 4. Markdown 内容中的图片引用方式

我建议内部 Markdown 使用这种格式：

```md
![图片](asset://image_01HXABC.png)
```

或者：

```md
![图片](edgenotes://asset/asset_id)
```

但浏览器/编辑器无法直接显示这种协议，所以渲染时需要转换。

实际保存：

```md
今天的灵感：

![截图](edgenotes://asset/01JXXX)
```

前端渲染时转换成：

```text
http://asset.localhost/xxx
```

或者 Tauri 的 asset protocol 地址。

更简单的第一版方案是：

```md
![截图](file:///C:/Users/xxx/AppData/Roaming/EdgeNotes/assets/images/2026/05/xxx.png)
```

但我不推荐长期用绝对路径，因为迁移和导出会麻烦。

更好的设计是：

```text
数据库里保存 asset_id → file_path
Markdown 里保存 edgenotes://asset/asset_id
编辑器显示时解析成真实可访问地址
导出时再转换成相对路径
```

# 7. 窗口核心设计

窗口状态：

```ts
type DrawerState =
  | "collapsed"
  | "expanding"
  | "expanded"
  | "editing"
  | "collapsing";
```

为什么要有 `editing`？

因为用户正在写 Markdown 时，鼠标可能短暂离开窗口，如果立刻收起，体验会炸。

逻辑：

```text
collapsed：
  只显示顶部小标签

expanded：
  展开便签抽屉

editing：
  编辑器获得焦点，不自动收起

collapsing：
  动画收起中
```

核心规则：

```text
鼠标进入标签 → expanded
编辑器 focus → editing
编辑器 blur + 鼠标离开 → 延迟收起
按 Esc → collapsed
按全局快捷键 → expanded / collapsed 切换
拖拽图片进入 → expanded 并锁定窗口
```

窗口尺寸：

```ts
const WINDOW_SIZE = {
  collapsed: {
    width: 260,
    height: 40,
  },
  expanded: {
    width: 860,
    height: 620,
  },
};
```

位置计算：

```text
x = 当前屏幕宽度 / 2 - 当前窗口宽度 / 2
y = 0
```

Rust 命令：

```rust
#[tauri::command]
fn set_window_top_center(app: tauri::AppHandle, width: f64, height: f64) {
    let window = app.get_webview_window("main").unwrap();

    if let Ok(Some(monitor)) = window.current_monitor() {
        let size = monitor.size();
        let scale_factor = monitor.scale_factor();

        let monitor_width = size.width as f64 / scale_factor;

        let x = (monitor_width - width) / 2.0;
        let y = 0.0;

        let _ = window.set_size(tauri::LogicalSize::new(width, height));
        let _ = window.set_position(tauri::LogicalPosition::new(x, y));
    }
}
```

前端封装：

```ts
import { invoke } from "@tauri-apps/api/core";

export async function setTopCenter(width: number, height: number) {
  return invoke("set_window_top_center", {
    width,
    height,
  });
}
```

# 11. 图片服务设计

`assetService.ts`：

```ts
export interface SavedAsset {
  id: string;
  noteId: string;
  fileName: string;
  mimeType: string;
  size: number;
  markdownUrl: string;
  previewUrl: string;
}

export async function saveImageAsset(
  file: File,
  noteId: string
): Promise<SavedAsset> {
  const arrayBuffer = await file.arrayBuffer();

  return invoke<SavedAsset>("save_image_asset", {
    noteId,
    fileName: file.name,
    mimeType: file.type,
    bytes: Array.from(new Uint8Array(arrayBuffer)),
  });
}
```

```rust
    file_name: String,
    mime_type: String,
    bytes: Vec<u8>,

    // 1. 校验 mime_type 是否是 image/*
    // 2. 生成 asset_id
    // 3. 生成保存路径
    // 4. 写入图片文件
    // 5. 写入 assets 表
    // 6. 返回 markdownUrl 和 previewUrl
```

# 12. 图片粘贴逻辑

前端处理：

```ts
export async function handlePasteImage(
  event: React.ClipboardEvent,
  noteId: string,
  insertMarkdown: (markdown: string) => void
) {
  const items = Array.from(event.clipboardData.items);

  const imageItem = items.find((item) => item.type.startsWith("image/"));

  if (!imageItem) {
    return;
  }

  event.preventDefault();

  const file = imageItem.getAsFile();

  if (!file) {
    return;
  }

  const asset = await saveImageAsset(file, noteId);

  insertMarkdown(`\n![${asset.fileName}](${asset.markdownUrl})\n`);
}
```

注意：粘贴图片时，很多截图工具给的文件名为空。你要自动生成：

```text
pasted-20260529-153012.png
```

# 13. 拖拽图片逻辑

```ts
export async function handleDropImage(
  event: React.DragEvent,
  noteId: string,
  insertMarkdown: (markdown: string) => void
) {
  event.preventDefault();

  const files = Array.from(event.dataTransfer.files);

  const images = files.filter((file) => file.type.startsWith("image/"));

  for (const image of images) {
    const asset = await saveImageAsset(image, noteId);

    insertMarkdown(`\n![${asset.fileName}](${asset.markdownUrl})\n`);
  }
}
```

# 17. 自动保存设计

Markdown 编辑器会频繁触发 `onChange`，不能每次都写 SQLite。

要做防抖：

```ts
export function useDebouncedSave(
  noteId: string,
  content: string,
  delay = 600
) {
  useEffect(() => {
    const timer = window.setTimeout(() => {
      noteService.updateNote(noteId, content);
    }, delay);

    return () => window.clearTimeout(timer);
  }, [noteId, content, delay]);
}
```

更完整一点：

```text
用户正在输入：状态显示 “正在编辑”
600ms 无输入：保存到 SQLite
保存成功：状态显示 “已保存”
保存失败：状态显示 “保存失败”
```

状态：

```ts
type SaveStatus =
  | "idle"
  | "editing"
  | "saving"
  | "saved"
  | "error";
```

UI 放在编辑器右上角：

```text
已保存
```

# 18. Markdown 标题提取

更新内容时自动生成标题。

规则：

```ts
export function extractTitle(markdown: string): string {
  const lines = markdown
    .split("\n")
    .map((line) => line.trim())
    .filter(Boolean);

  if (lines.length === 0) {
    return "未命名便签";
  }

  const first = lines[0];

  return first
    .replace(/^#{1,6}\s+/, "")
    .replace(/^[-*+]\s+/, "")
    .replace(/!\[.*?\]\(.*?\)/g, "[图片]")
    .slice(0, 60);
}
```

摘要提取：

```ts
export function extractExcerpt(markdown: string): string {
  return markdown
    .replace(/!\[.*?\]\(.*?\)/g, "[图片]")
    .replace(/[#>*_`~\-]/g, "")
    .replace(/\n+/g, " ")
    .trim()
    .slice(0, 120);
}
```

```text

```

# 24. 图片保存 Rust 逻辑

伪代码：

```rust
fn save_image_asset_impl(
    app: &AppHandle,
    note_id: String,
    original_name: String,
    mime_type: String,
    bytes: Vec<u8>,
) -> Result<SavedAsset, String> {
    if !mime_type.starts_with("image/") {
        return Err("不是图片文件".into());
    }

    let asset_id = generate_id();
    let ext = mime_to_ext(&mime_type)?;
    let now = chrono::Utc::now();

    let relative_path = format!(
        "assets/images/{}/{}/{}.{}",
        now.format("%Y"),
        now.format("%m"),
        asset_id,
        ext
    );

    let full_path = app_data_dir.join(&relative_path);

    create_dir_all(parent_dir)?;
    write(&full_path, bytes)?;

    insert_asset_to_db(
        asset_id,
        note_id,
        original_name,
        relative_path,
        mime_type,
        size,
    )?;

    Ok(SavedAsset {
        id: asset_id.clone(),
        file_name: original_name,
        markdown_url: format!("edgenotes://asset/{}", asset_id),
        preview_url: make_preview_url(asset_id),
    })
}
```

# 26. Markdown 加载与保存的转换

因为你的 Markdown 内部可能是：

```md

```

但编辑器展示需要真实图片地址。

所以要有两个转换函数：

```ts
export async function markdownToEditorMarkdown(markdown: string) {
  // edgenotes://asset/id
  // ↓
  // 可显示的图片 URL
}

export async function editorMarkdownToStorageMarkdown(markdown: string) {
  // 可显示图片 URL
  // ↓
  // edgenotes://asset/id
}
```

例子：

```ts
export async function markdownToEditorMarkdown(markdown: string) {
  const assetPattern = /edgenotes:\/\/asset\/([a-zA-Z0-9_-]+)/g;

  let result = markdown;
  const matches = [...markdown.matchAll(assetPattern)];

  for (const match of matches) {
    const assetId = match[1];
    const previewUrl = await assetService.resolveAssetUrl(assetId);

    result = result.replace(match[0], previewUrl);
  }

  return result;
}
```

保存前反向转换：

```ts

}
```

# 27. 图片删除与垃圾回收

用户删除 Markdown 里的图片时，图片文件不会自动删除。

你需要一个资源清理机制：

```text
保存便签时，扫描 Markdown 中仍在使用的 asset_id
查询该 note_id 下所有 assets
不再被引用的 asset 标记为 orphan
定期清理 orphan 图片
```

assets 表加字段：

```sql
ALTER TABLE assets ADD COLUMN orphaned INTEGER NOT NULL DEFAULT 0;
```

逻辑：

```text
每次保存 note：
  usedAssetIds = parseAssetIds(markdown)
  noteAssets = listAssets(noteId)
  for asset in noteAssets:
    if asset.id not in usedAssetIds:
      mark orphaned = 1
```

不要立即删除，因为用户可能 Ctrl+Z 撤回。

可以 7 天后清理：

```text
orphaned = 1 且 updated_at 超过 7 天 → 删除文件 + 删除数据库记录
```

# 34. 最终产品的编码核心

你这个项目最核心的代码不是 Markdown，也不是 SQLite，而是这 4 个闭环：

```text
窗口闭环：
collapsed ↔ expanded ↔ editing ↔ collapsed

编辑闭环：
Markdown 输入 → 防抖保存 → SQLite → 重新加载

图片闭环：
选择/粘贴/拖拽图片 → 复制到 appData → assets 表 → Markdown 引用 → 编辑器显示

资源闭环：
Markdown 扫描 asset_id → 判断图片是否仍被使用 → 标记孤儿资源 → 定期清理
```

只要这 4 个闭环稳定，你这个项目就不是玩具，而是一个真正可长期迭代的软件。

我的最终建议是：**编辑器直接选 Milkdown/Crepe，图片不要进数据库，SQLite 只存 Markdown 和资源索引，所有图片复制到应用数据目录，Markdown 内部用 `edgenotes://asset/id` 引用。** 这个架构后面加导出、同步、AI 总结、标签、搜索都不会推倒重来。
